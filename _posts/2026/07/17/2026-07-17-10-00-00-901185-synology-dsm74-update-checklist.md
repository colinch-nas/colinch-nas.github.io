DSM 7.4가 2026년 6월 16일 공개됐다. AI 기반 관리와 저장 효율 개선이 눈에 띄지만, DS923+에서 업데이트 버튼부터 누르는 건 피하는 게 좋다. 실제로는 패키지 호환성, 공유 폴더 스냅샷, 외부 접속 설정이 정상인지 확인하는 데 시간이 더 걸린다. 아래 순서대로 점검하면 업데이트 후 “파일은 있는데 앱이 안 뜨는” 상황을 줄일 수 있다.

그림에서 볼 부분은 NAS 본체보다 업데이트 전후에 확인해야 할 저장소와 서비스 흐름이다.

## 업데이트 전 환경

내가 기준으로 삼은 환경은 DS923+, DSM 7.3 계열, SHR-1(Btrfs), Container Manager로 운영하는 홈서버다. DSM 7.4는 모델별 제공 시점과 지원 기능이 다르므로 제어판의 업데이트 알림을 기준으로 판단해야 한다. Synology도 이번 버전을 AI 관리, 협업, 저장 효율 개선 릴리스로 소개하고 있다.

| 확인 항목 | 업데이트 전 기록할 값 | 이유 |
|---|---|---|
| DSM | 7.3.x | 롤백 필요 시 기준점 |
| 저장소 | Btrfs, 볼륨 여유 공간 | 변환·복구 시간 판단 |
| 패키지 | Container Manager, Hyper Backup | 중단 서비스 확인 |
| 외부 접속 | Tailscale 또는 역방향 프록시 | 업데이트 후 재접속 시험 |

## 실제 점검 순서

### 1. 설정과 패키지 목록을 백업한다

제어판 → 업데이트 및 복원 → 구성 백업에서 DSM 설정을 파일로 내려받는다. Hyper Backup 작업은 실행 중이지 않은 시간에 돌리고, Container Manager 프로젝트의 Compose 파일도 별도로 복사한다.

```bash
# SSH를 켠 경우 프로젝트 정의만 별도 보관한다
sudo cp -a /volume1/docker /volume1/backup/docker-before-dsm74
sudo docker compose ls
```

`docker` 폴더 전체를 백업한다고 컨테이너 데이터베이스가 항상 일관되게 저장되는 것은 아니다. MariaDB나 PostgreSQL을 쓰는 서비스는 앱의 내장 백업 또는 DB 덤프를 함께 남겨야 한다.

### 2. 저장소 상태를 보고 업데이트한다

저장 관리자에서 드라이브 경고, 스토리지 풀 상태, 볼륨 여유 공간을 확인한다. 여유 공간이 10% 아래면 업데이트보다 정리가 우선이다. 업데이트 후 저장 효율 기능이 보이더라도 호환 드라이브와 볼륨 조건에 따라 메뉴가 다르게 나타날 수 있다.

### 3. 재부팅 후 서비스별로 확인한다

자동으로 전부 복구될 것이라 생각했는데, 실제 점검에서는 패키지는 실행 중이어도 공유 폴더 권한이나 USB 백업 작업이 멈춰 있는 경우가 더 골치 아팠다. 아래 주소를 직접 열어 본다.

| 서비스 | 확인 방법 | 실패 시 볼 곳 |
|---|---|---|
| SMB 공유 | PC에서 파일 생성·삭제 | 공유 폴더 권한, ACL |
| Container Manager | 프로젝트 상태가 실행 중인지 확인 | 프로젝트 로그 |
| Hyper Backup | 최근 작업 성공 여부 | 작업 로그와 대상 디스크 |
| Tailscale | NAS의 VPN 주소로 DSM 접속 | 패키지 상태, 노드 승인 |

### 4. 보안 설정을 다시 확인한다

제어판 → 보안 → 보호에서 자동 차단 목록과 방화벽 규칙을 확인한다. 관리자 계정으로 계속 사용하고 있다면 일반 사용자 계정을 만들고 관리자 계정은 비활성화하는 편이 낫다. DSM 업데이트가 외부 공개 포트를 대신 닫아 주지는 않는다. 라우터의 5000·5001 포트포워딩은 정말 필요한지 다시 판단한다.

## 업데이트를 미뤄도 되는 경우

| 상황 | 판단 |
|---|---|
| NAS가 파일 공유만 제공 | 공지된 호환성 확인 후 진행 |
| Docker에 데이터베이스 서비스가 많음 | 백업 복구 시험 후 진행 |
| 업무 시간에 사용하는 NAS | 주말 점검 창구에서 진행 |
| 모델별 DSM 7.4 알림이 아직 없음 | 수동 설치를 서두르지 않음 |

DSM 7.4는 새 기능만 보고 설치할 버전이 아니다. 설정 백업, 저장소 상태, 컨테이너와 백업 작업, 외부 접속을 한 바퀴 확인한 뒤 업데이트하고, 재부팅 후 같은 항목을 다시 확인하는 것이 안전하다.

짧게 정리하면 다음과 같다.

- DSM 설정과 Compose 파일을 따로 보관한다.
- 저장소 여유 공간과 Btrfs 상태를 확인한다.
- 재부팅 후 SMB·Container Manager·Hyper Backup을 직접 시험한다.
- DSM 업데이트와 포트포워딩 해제는 별개의 작업이다.
