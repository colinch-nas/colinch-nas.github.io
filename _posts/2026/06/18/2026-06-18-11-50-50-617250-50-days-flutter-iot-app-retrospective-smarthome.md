# 50일간의 Flutter IoT 앱 개발 회고 - 스마트홈을 만들며

![스마트홈 완성](https://images.unsplash.com/photo-1558618666-fcd25c85cd64?w=800&q=80)

## 돌아보며

50편을 쓰면서 프로젝트 전체를 다시 돌아봤다. 개발할 때는 하루하루 눈앞의 문제를 해결하느라 바빴는데, 정리해보니 생각보다 많은 걸 만들었다는 걸 새삼 느꼈다.

BLE 프로토콜부터 시작해서 MQTT, Realm, Firebase, 다국어 지원, 다크모드, 차트, 알림, SCADA, FOTA까지. 처음 프로젝트를 받았을 때 "이게 다 Flutter로 가능한가?" 싶었는데, 해보니 됐다.

## 가장 어려웠던 부분

단연코 **BluFi 프로토콜 구현**이었다. Flutter(Dart)로 구현된 레퍼런스가 없어서 Espressif의 Android SDK 소스를 분석해서 이식했다. 바이트 레벨 조작, CRC 체크섬 계산, 패킷 분할과 조립... 디버깅 환경도 열악해서 Serial 로그와 Wireshark로 패킷을 직접 분석했다.

그다음으로 어려웠던 건 **AWS IoT Core SigV4 인증**이었다. MQTT over WebSocket URL에 서명을 붙이는 방식인데, 서명 방법이 정확하지 않으면 `403 Forbidden`만 나와서 원인 파악이 쉽지 않았다.

## Flutter IoT 앱 개발의 장단점

솔직하게 정리해보면:

**장점**:
- 한 코드베이스로 Android/iOS 동시 개발 → 리소스 절약이 확실함
- Hot Reload로 UI 개발 속도가 매우 빠름
- BLE, MQTT, Firebase 등 주요 패키지들이 충분히 성숙함
- Dart 언어가 생각보다 쓸 만하다. 특히 null safety가 런타임 오류를 많이 줄여줌

**단점**:
- BLE 관련 Android/iOS 차이를 Flutter 레이어에서 다 흡수하기 어렵다. 결국 플랫폼별 코드를 건드려야 하는 경우가 있음
- 앱 크기가 크다. 순수 네이티브보다 Flutter Runtime 때문에 기본 크기가 큼
- 복잡한 커스텀 UI는 네이티브만큼 구현하기 까다롭다

## GetX + Clean Architecture 조합에 대한 솔직한 평가

GetX는 쓰기 쉽고 빠르다. 근데 프로젝트가 커지면 Controller가 비대해지는 경향이 있다. 규율 없이 쓰면 모든 게 Controller에 들어가게 된다.

Clean Architecture로 레이어를 강제로 분리해두니 Controller에 UI 상태만 남고, 실제 로직은 Repository와 Service로 빠지게 되어 관리가 훨씬 편했다.

이 조합을 다시 쓸 거냐고 묻는다면 "예스"다. 단, GetX의 마법 같은 편의성에 의존하지 않고 구조를 명확히 잡는 게 전제조건이다.

## 다음 버전에서 개선하고 싶은 것들

**1. 테스트 코드**: 이번 프로젝트에서 Unit Test를 충분히 못 썼다. Repository 패턴 덕분에 Mock 교체는 쉬운데, 실제 테스트 코드가 없다. 다음 버전에서는 Repository 레이어부터 테스트를 작성하고 싶다.

**2. 에러 처리 일관성**: 에러 처리 방식이 화면마다 조금씩 다르다. `Get.snackbar()`를 쓰는 곳도 있고, 화면 내 에러 텍스트를 쓰는 곳도 있고. 일관된 에러 처리 컨벤션을 잡고 싶다.

**3. 오프라인 지원 강화**: 현재는 "마지막 상태 보여주기" 수준이다. 오프라인 중 변경한 설정을 나중에 동기화하는 진정한 오프라인 지원을 구현하고 싶다.

## 다른 개발자들에게 추천하고 싶은 것들

IoT 앱을 Flutter로 개발하려는 분들께 몇 가지를 추천한다.

**BLE 구현은 일찍 시작해라**: BLE는 Android/iOS 차이가 크고, 실제 기기 없이는 테스트가 어렵다. 프로젝트 초반에 BLE 연결 기본 흐름을 확인해두는 게 좋다.

**MQTT 토픽 설계는 신중하게**: 처음에 잘 못 잡으면 나중에 바꾸기 정말 힘들다. 버전, 기기 타입, 기기 ID를 모두 포함하는 구조를 처음부터 잡아라.

**테스트 모드를 미리 만들어라**: 실제 기기가 없어도 개발할 수 있는 Mock 구현체를 초반에 만들어두면 팀 전체 개발 속도가 올라간다.
